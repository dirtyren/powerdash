# P2.3 — Widget-Type Palette (20 echarts kinds) Design

**Status:** Design approved 2026-04-29 — ready for implementation plan.
**Depends on:** P2.2 (create-dashboard + palette infrastructure)
**Estimate:** ~40 engineering hours

---

## 1. Goal

Replace the P2.2 fixture-style catalog with a **chart-type palette** covering 20 echarts kinds. Each palette entry represents a widget TYPE; clicking drops a fresh instance onto the canvas with a generated id, default title, default size, and built-in sample data. Multiple instances of the same type are expected. The three existing P1 fixture widgets on dashboard 1 continue to fetch from WireMock and render as before.

## 2. Scope

**In scope**
- 20 widget kinds (section 3).
- Generic `<EchartsWidget>` renderer for the 18 echarts-backed kinds.
- Per-kind **adapter** module with default dimensions, sample data, and option builder (or custom Renderer for kpi/table).
- `WIDGET_ADAPTERS` registry keyed by `WidgetKind`.
- Family-grouped, scrollable `<WidgetPalette>`.
- Inline widget-title rename on the editable canvas.
- 404 sample-data fallback for kpi/line/table when fetch misses.
- Breaking-change cleanup of P2.2 palette plumbing: drop catalog-id duplicate guard, drop `existingWidgetIds` prop, remove `widget-catalog.ts`.

**Out of scope**
- Real backend data for fresh widgets (sample only).
- User-uploaded data, dataset management.
- Per-widget customization beyond title (axis labels, colors, series edit, thresholds).
- Drag-widget-between-dashboards.
- Widget deletion (already available; not touched).
- Dynamic catalog from backend — `WIDGET_ADAPTERS` stays hard-coded.

## 3. The 20 kinds

| Family | Kinds |
|---|---|
| `stats` | `kpi`, `gauge` |
| `series` | `line`, `area`, `bar`, `stacked-bar`, `scatter` |
| `comparison` | `radar` |
| `part-of-whole` | `pie`, `donut`, `funnel` |
| `hierarchy` | `tree`, `sunburst`, `treemap` |
| `density` | `heatmap`, `sankey` |
| `distribution` | `histogram`, `box-plot` |
| `financial` | `candlestick` |
| `data` | `table` |

`WidgetKindSchema` extends from 3 values to 20. No other schema changes.

## 4. Data model changes

- `src/server/schemas/widget.ts` — `WidgetKindSchema` expands:
  ```ts
  export const WidgetKindSchema = z.enum([
    "kpi", "gauge",
    "line", "area", "bar", "stacked-bar", "scatter",
    "radar",
    "pie", "donut", "funnel",
    "tree", "sunburst", "treemap",
    "heatmap", "sankey",
    "histogram", "box-plot",
    "candlestick",
    "table",
  ]);
  ```
- `WidgetDataSchema` (the discriminated union of KPI/Line/Table data) is **unchanged**. It still serves the 3 fixture widgets. The 17 new kinds don't round-trip through it — their data is inline in the adapter.
- `WidgetRef` shape unchanged. The only functional difference is that `kind` now accepts 20 values.
- New widget ids: `crypto.randomUUID().slice(0, 8)` (hex). Legacy `w-cpu-*` / `w-hosts-*` ids keep working alongside.

## 5. Adapter interface

New module `src/widgets/adapter.ts`:

```ts
import type { EChartsOption } from "echarts";
import type { WidgetKind, WidgetRef, WidgetData } from "@/server/schemas/widget";
import type { ComponentType } from "react";

export type WidgetFamily =
  | "stats" | "series" | "comparison" | "part-of-whole"
  | "hierarchy" | "density" | "distribution" | "financial" | "data";

export interface WidgetAdapter {
  kind: WidgetKind;
  family: WidgetFamily;
  displayName: string;           // human-readable, e.g., "Bar chart"
  defaultTitle: string;
  defaultW: number;
  defaultH: number;
  // Exactly one of:
  buildOption?: (widget: WidgetRef) => EChartsOption;   // echarts-backed kinds
  Renderer?: ComponentType<{ widget: WidgetRef }>;      // kpi, table
  sampleData?: WidgetData;       // used by kpi/line/table on 404 fallback
}

export const WIDGET_ADAPTERS: Record<WidgetKind, WidgetAdapter>;
```

One adapter module per kind lives under `src/widgets/adapters/<kind>.ts`. Each is ~15–25 lines.

**Echarts adapter (example: bar):**
```ts
import type { WidgetAdapter } from "../adapter";

export const barAdapter: WidgetAdapter = {
  kind: "bar",
  family: "series",
  displayName: "Bar chart",
  defaultTitle: "Bar chart",
  defaultW: 480,
  defaultH: 320,
  buildOption: () => ({
    xAxis: { type: "category", data: ["A", "B", "C", "D", "E"] },
    yAxis: { type: "value" },
    series: [{ type: "bar", data: [12, 34, 23, 45, 30] }],
  }),
};
```

**Custom-renderer adapter (kpi):**
```ts
export const kpiAdapter: WidgetAdapter = {
  kind: "kpi",
  family: "stats",
  displayName: "KPI",
  defaultTitle: "KPI",
  defaultW: 260,
  defaultH: 160,
  Renderer: KpiTile,   // existing component (title-stripped per §7)
  sampleData: { kind: "kpi", value: 42, unit: "%", delta: 3.1 },
};
```

**Registry completeness check:** `WIDGET_ADAPTERS` is typed as `Record<WidgetKind, WidgetAdapter>`, which forces TypeScript to fail compilation if any kind is missing. A unit test iterates every `WidgetKindSchema` value and asserts `WIDGET_ADAPTERS[kind]` exists with `kind === key`.

## 6. Rendering

**New component: `src/components/widgets/EchartsWidget.tsx`** — thin wrapper around `echarts-for-react` (already imported via `next/dynamic` lazy factory). Takes `{ option: EChartsOption }` and renders. No title handling (that's the frame's job, §7).

**New component: `src/components/widgets/WidgetFrame.tsx`** — the title bar + border + hover affordances for every widget. Replaces the per-component title rendering in `KpiTile` / `LineChart` / `DataTable`:

```tsx
interface Props {
  title: string;
  onTitleChange?: (next: string) => void;  // if present, title is editable inline
  children: ReactNode;
}
```

Behavior: when `onTitleChange` is provided, the title is rendered as an `<input aria-label="Widget title" />` that calls `onTitleChange` on change (blur/Enter commit is implicit since React state updates on every change — matches the EditToolbar pattern from P2.2 §7). When absent, renders a plain `<span>` — matches the read-only canvas.

**Canvas wiring:**
- `DashboardCanvas.tsx` (read-only): wraps each widget in `<WidgetFrame title={w.title}>`. Inside the frame, `WidgetByKind` looks up the adapter and renders either `<adapter.Renderer widget={w} />` or `<EchartsWidget option={adapter.buildOption!(w)} />`.
- `EditableDashboardCanvas.tsx`: same, but `<WidgetFrame title={w.title} onTitleChange={(next) => update(w.id, { title: next })}>`. The `update` helper already exists.

**Adapter resolution (shared):**
```ts
function WidgetByKind({ widget }: { widget: WidgetRef }) {
  const adapter = WIDGET_ADAPTERS[widget.kind];
  if (adapter.Renderer) return <adapter.Renderer widget={widget} />;
  return <EchartsWidget option={adapter.buildOption!(widget)} />;
}
```

## 7. Component refactors (kpi/line/table)

These three existing components currently render their own title AND fetch their own data. Two changes:

1. **Strip title rendering.** Their `title` prop is removed. They emit only the content (number + unit for KPI, chart for line, table for data). The WidgetFrame now owns the title.
2. **404 fallback to `sampleData`.** Each component uses a small helper:
   ```ts
   function useWidgetDataOrSample<T extends WidgetData>(
     widgetId: string,
     fallback: T,
   ): T {
     const { data, isError } = useWidgetData(widgetId);
     if (isError) return fallback;
     return (data as T | undefined) ?? fallback;
   }
   ```
   The adapter's `sampleData` is the fallback. On the fixture dashboard (ids `w-cpu-*` / `w-hosts-*`), WireMock returns real data and fallback never fires. For fresh widgets (UUID ids with no fixture), the 404 path triggers and the sample renders.

**Props after refactor:**
- `<KpiTile widget={w}>` — receives the whole `WidgetRef`.
- `<LineChart widget={w}>` — same.
- `<DataTable widget={w}>` — same.

This matches the adapter `Renderer: ComponentType<{ widget: WidgetRef }>` contract.

## 8. Palette refactor

**`<WidgetPalette>` (rewrite):**

```tsx
export function WidgetPalette({ onAdd }: { onAdd: (adapter: WidgetAdapter) => void }) {
  const grouped = useMemo(() => groupByFamily(Object.values(WIDGET_ADAPTERS)), []);
  return (
    <aside className="w-64 overflow-y-auto border-l border-border bg-card p-4">
      <h2 className="mb-4 text-sm font-semibold uppercase tracking-wide text-muted-foreground">
        Widgets
      </h2>
      {FAMILY_ORDER.map((family) => (
        <section key={family} className="mb-4">
          <h3 className="mb-2 text-xs uppercase text-muted-foreground">{familyLabel(family)}</h3>
          <ul className="flex flex-col gap-1">
            {grouped[family].map((adapter) => (
              <li key={adapter.kind}>
                <button
                  type="button"
                  onClick={() => onAdd(adapter)}
                  className="flex w-full items-center justify-between rounded border border-border bg-background px-3 py-1.5 text-left text-sm hover:bg-muted"
                >
                  <span>{adapter.displayName}</span>
                </button>
              </li>
            ))}
          </ul>
        </section>
      ))}
    </aside>
  );
}
```

**Changes from P2.2:**
- `catalog` and `existingWidgetIds` props removed.
- `onAdd` takes a `WidgetAdapter`.
- No duplicate-guard (buttons are always enabled — multiple instances are intentional).
- Family grouping with fixed display order: stats, series, comparison, part-of-whole, hierarchy, density, distribution, financial, data.
- Scrollable (`overflow-y-auto`) since 20 entries exceed the drawer height at most viewports.

**Consumers updated:**
- `/dashboards/new` and `/[id]/edit`:
  - Drop the `existingWidgetIds` computation.
  - `handleAddWidget(adapter)` now: `const id = crypto.randomUUID().slice(0, 8); const next: WidgetRef = { id, kind: adapter.kind, title: adapter.defaultTitle, x: 20, y: 20, w: adapter.defaultW, h: adapter.defaultH };`

**Deletion:** `src/config/widget-catalog.ts` and `tests/unit/widget-catalog.test.ts` are removed. Superseded by `src/widgets/adapter.ts` + per-kind registry tests.

## 9. Bundle discipline for echarts

The 18 echarts-backed kinds will pull in several chart modules. To avoid shipping the whole ~1 MB library:

- Do NOT `import echarts from "echarts"` anywhere.
- `EchartsWidget.tsx` imports only the **core**:
  ```ts
  import * as echarts from "echarts/core";
  import { CanvasRenderer } from "echarts/renderers";
  echarts.use([CanvasRenderer]);
  ```
- Each adapter registers its required chart + coordinate + component modules:
  ```ts
  // src/widgets/adapters/bar.ts
  import * as echarts from "echarts/core";
  import { BarChart } from "echarts/charts";
  import { GridComponent, TooltipComponent } from "echarts/components";
  echarts.use([BarChart, GridComponent, TooltipComponent]);
  ```
- `echarts-for-react` picks up the globally-registered modules.

This makes the set of loaded modules **additive** — each kind contributes what it needs. Adapters that share components (e.g., all the series kinds need `GridComponent`) don't double-register.

## 10. Tests

**Unit (Vitest):**
- `adapter.test.ts` — iterate every `WidgetKindSchema` value; assert `WIDGET_ADAPTERS[kind].kind === key`; assert exactly one of `buildOption` or `Renderer` is set.
- `adapters.bar.test.ts` (one per echarts kind) — tiny: call `buildOption(sampleWidget)`, assert the returned option has the expected `series[0].type` and non-empty `series[0].data`. Keep each test ≤ 5 lines; 18 files.
- `WidgetFrame.test.tsx` — renders title as `<span>` without callback; as `<input>` with callback; fires `onTitleChange` on change; existing children render.
- `EchartsWidget.test.tsx` — renders without crashing given a minimal option (snapshot the `role` landmark — the actual chart doesn't render in happy-dom).
- `WidgetPalette.test.tsx` — replaces P2.2 tests. New coverage: family sections render in fixed order; every kind produces a button; clicking fires `onAdd` with the adapter; no duplicate guard (clicking twice fires twice).
- `KpiTile.test.tsx` / `LineChart.test.tsx` / `DataTable.test.tsx` — update existing tests: now receive `widget` prop; on fetch error, render `sampleData`.

**E2E (Playwright):**
- `dashboard-create.spec.ts` — update selectors from `/CPU %/` (fixture-specific) to `/KPI/` (generic type label).
- New spec: `dashboard-widget-types.spec.ts` — on `/dashboards/new`, add a Bar, a Pie, and a Gauge; rename one inline; save; assert redirect and that the saved dashboard (re-GET returns fixture 1 regardless, so assert only that the save request happened and the page lands on `/dashboards/1`).

## 12. Risks

1. **Echarts types sprawl.** `EChartsOption` varies per chart family. Adapters will use typed option helpers from `echarts/types` rather than a loose `Record<string, unknown>`. If the types fight us on exotic kinds (Sankey, Tree), we narrow per-kind.
2. **Bundle size.** Without per-kind tree-shaking (§9), we'd ship ~1 MB of echarts. With explicit imports in each adapter, bundle grows additively. Acceptance check during implementation: `next build` output shows reasonable first-load JS.
3. **Visual correctness.** 20 kinds = 20 places to get sample data wrong. "Visual pass" in the estimate is where each kind gets opened in the browser and eyeballed. Expect 2–3 kinds to need a second pass.
4. **404-fallback is structural.** Using HTTP error to pick sample-vs-real data couples fetch state to render. Fine for MVP; deserves a cleanup when real backend data lands.
5. **Scope creep.** At 20 kinds the temptation to add per-kind customization (colors, thresholds, axis labels) is real. None of that is in scope. Hold the line.
6. **Perf with many charts.** A busy dashboard may hitch on resize. Already lazy-loaded. If it bites, virtualize by visibility later.
