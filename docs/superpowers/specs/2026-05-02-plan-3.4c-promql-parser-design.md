# P3.4c — Bidirectional PromQL parser (Code → Builder) Design

**Status:** Design approved 2026-05-02 — ready for implementation plan.
**Depends on:** P3.4a (`@prometheus-io/lezer-promql` installed), P3.4b (`BuilderState`, `buildExpression`, `<QueryBuilder>`, `<ModeTabs>`, `<QueryEditor>` Code/Builder refactor).
**Estimate:** ~14 engineering hours (~1.75 days).

This closes out the PromQL arc for the dashboard. P3.4b's builder was one-way (form → code). P3.4c adds the reverse: when a user has a PromQL expression in Code and clicks the Builder tab, we parse the expression into `BuilderState` and populate the form. Non-builder-shape expressions keep the Builder tab disabled with a tooltip.

---

## 1. Goal

Flip the `<QueryEditor>` Builder tab from "always enabled, resets form to empty" to "enabled iff the current expression can be represented; populated from the parse." Non-builder-shape expressions (binary ops, subqueries, scalar arithmetic, unsupported functions, etc.) disable the Builder tab with a native `title` tooltip that explains why.

## 2. Scope

**In scope**
- Pure function `parseExpression(code: string): BuilderState | null`. Uses the Lezer grammar from `@prometheus-io/lezer-promql` (installed in P3.4a). Accepts exactly what `buildExpression` emits; rejects everything else with `null`.
- `<ModeTabs>` gains an optional `builderDisabled?: { reason: string }` prop — when set, the Builder tab is rendered `disabled` with a `title` tooltip.
- `<QueryEditor>` runs `parseExpression(expr)` via `useMemo`, sets `canBuilder = expr.trim() === "" || parsed !== null`, and passes the disabled affordance to `<ModeTabs>` when `!canBuilder`.
- On Code → Builder switch, populate `builderState` from the parse result.
- Safeguard effect: if the editor flips into Builder and the expression becomes unparseable, force back to Code.
- Unit tests: 20 parser cases including error gates, shape coverage, round-trip property on arbitrary `BuilderState`s, and node-name verification. ModeTabs tests add a `builderDisabled` case. QueryEditor tests add parseable-vs-unparseable tab behavior.

**Out of scope** (deferred)
- Partial parse with warnings (populate the builder from the parseable sub-tree of `sum(up) / 2`).
- Preserving string-level round-trip (single-quote ↔ double-quote, extra whitespace). We accept semantic equivalence only.
- Supporting binary operators, subqueries, `offset` / `@` modifier, scalar math, `topk / bottomk / quantile(k, ...)`, or any function-wrap other than `rate / irate`. These remain Code-mode-only.
- Debouncing the parser on keystroke. Runs synchronously in `useMemo`; Lezer is fast enough.

## 3. Grammar surface

The walker recognizes exactly three outer shapes, matching `buildExpression` composition (`aggregation( rate( metric{filters}[interval] ) by|without (labels) )`). Anything else returns `null`.

### 3.1 Top-level shape dispatch

At the first meaningful node (after `PromQL` / `Expr` wrappers), dispatch on `node.type.name`:

| Outer node | Walker step |
|---|---|
| `AggregateExpr` | Parse aggregation; recurse into its inner argument for rate/selector. |
| `FunctionCall` | Must be `rate` or `irate`. Parse rate wrap; recurse into its `MatrixSelector` argument. |
| `VectorSelector` | Base case — parse metric name and label matchers. |
| Anything else (`BinaryExpr`, `SubqueryExpr`, `NumberLiteral`, `FunctionCall` with unsupported name, `ParenExpr`, ...) | Return `null`. |

### 3.2 `AggregateExpr`

Extract:
- `AggregateOp` text — must be in the whitelist `{sum, avg, max, min, count, stddev, stdvar}`. Anything else (notably `topk`, `bottomk`, `quantile`, `count_values`, `group`) → `null`.
- `AggregateModifier` (optional child):
  - Absent → `groupKind: "none"`, `groupLabels: []`.
  - Present with `by` keyword → `groupKind: "by"`.
  - Present with `without` keyword → `groupKind: "without"`.
  - `GroupingLabels` children → the label names.
- The single argument expression — recurse into it. That recursion must yield a `BuilderState` with `aggregation` left unset (we set it here). If the recursion returns `null` or a state that already has `aggregation`, return `null` (rejects nested aggregations).

### 3.3 `FunctionCall` (rate / irate)

Extract:
- `FunctionIdentifier` — must be `rate` or `irate`. Otherwise `null`.
- `FunctionCallBody` → single argument expression — must be a `MatrixSelector`. Otherwise `null`.
- `MatrixSelector`'s `Duration` child → the interval string (e.g., `"5m"`).
- `MatrixSelector`'s `VectorSelector` child → recurse. The returned state must have `rate` unset and `aggregation` unset; we set `rate` here. If either field is already set, return `null` (nested rate or rate-inside-rate).

### 3.4 `VectorSelector`

Extract:
- `MetricIdentifier` — **required**. If absent (filters-only shape `{job="api"}`), return `null`.
- `LabelMatchers` (optional) → zero or more `LabelMatcher` children:
  - `LabelName` text.
  - `MatchOp` text — one of `= != =~ !~`.
  - `StringLiteral` — the value, unquoted via `unquote()` helper (handles both `"..."` and `'...'`, decodes escapes).

Return `{metric, filters, (no rate, no aggregation)}`.

### 3.5 `unquote` helper

```ts
function unquote(raw: string): string {
  if (raw.length >= 2 && raw.startsWith('"') && raw.endsWith('"')) {
    return JSON.parse(raw);
  }
  if (raw.length >= 2 && raw.startsWith("'") && raw.endsWith("'")) {
    // Convert single-quoted to double-quoted for JSON.parse.
    const inner = raw.slice(1, -1).replace(/\\'/g, "'").replace(/"/g, '\\"');
    return JSON.parse('"' + inner + '"');
  }
  return raw;
}
```

Escape sequences inside `"..."` are decoded via `JSON.parse` (which handles `\"`, `\\`, `\n`, `\t`, `\uXXXX`). `'..'` strings have `\'` unescaped first, then any `"` is escaped, and the result goes through `JSON.parse`. PromQL allows backtick strings too (raw), but the builder never emits them; treat them as unsupported → fall through to the `return raw` branch, which will likely fail a later sanity check and the whole parse returns `null`. Acceptable.

## 4. Parser module

`src/widgets/promql/parse-expression.ts` — all internal helpers private to the module, only `parseExpression` exported:

```ts
import { parser as promQLParser } from "@prometheus-io/lezer-promql";
import type { SyntaxNode, Tree } from "@lezer/common";
import type {
  BuilderState,
  LabelFilter,
  LabelOp,
  AggregationFn,
  GroupKind,
} from "./builder-state";

const AGG_FNS: ReadonlySet<string> = new Set([
  "sum", "avg", "max", "min", "count", "stddev", "stdvar",
]);
const LABEL_OPS: ReadonlySet<string> = new Set(["=", "!=", "=~", "!~"]);

export function parseExpression(code: string): BuilderState | null {
  const trimmed = code.trim();
  if (!trimmed) return null;

  const tree = promQLParser.parse(trimmed);
  if (treeHasError(tree)) return null;

  const top = firstMeaningfulNode(tree.topNode);
  if (!top) return null;

  return parseTop(trimmed, top);
}

// ... internal helpers ...
```

Each helper function takes `(code: string, node: SyntaxNode)` and returns either a `BuilderState` or `null`. Total file size ~150 lines including tests-friendly error handling.

Exact node-name verification happens in **Task 1 of the plan** before implementation: the implementer dumps the AST for a handful of known inputs (`up`, `rate(up[5m])`, `sum(up) by (job)`) and confirms the node names match this design. If they differ (e.g., `AggregateOp` → `AggOp`), adjust the walker constants. Treat that as a small mechanical adjustment, not a plan rewrite.

## 5. `<ModeTabs>` extension

Add the optional prop:

```tsx
interface Props {
  mode: "code" | "builder";
  onChange: (m: "code" | "builder") => void;
  builderDisabled?: { reason: string };
}
```

Builder button when `builderDisabled` is set:
- Add `disabled={true}` attribute — the browser auto-prevents click.
- Add `title={builderDisabled.reason}` — native tooltip.
- Add `opacity-50 cursor-not-allowed` to the existing `className`.
- `aria-disabled={true}` for a11y.

The button's `onClick` doesn't need a guard — `<button disabled>` swallows clicks.

## 6. `<QueryEditor>` changes

New imports:
```tsx
import { parseExpression } from "@/widgets/promql/parse-expression";
```

Compute derived state just below the existing `useState`s:
```tsx
const parsedBuilderState = useMemo(
  () => parseExpression(expr),
  [expr],
);
const canBuilder = expr.trim() === "" || parsedBuilderState !== null;
```

Mode-change handler that populates on Code → Builder:
```tsx
const handleModeChange = (next: "code" | "builder") => {
  if (next === "builder" && parsedBuilderState) {
    setBuilderState(parsedBuilderState);
  }
  setMode(next);
};
```

Safeguard effect to snap back to Code when builder becomes unreachable:
```tsx
useEffect(() => {
  if (mode === "builder" && !canBuilder) setMode("code");
}, [mode, canBuilder]);
```

JSX change — replace `<ModeTabs mode={mode} onChange={setMode} />` with:
```tsx
<ModeTabs
  mode={mode}
  onChange={handleModeChange}
  builderDisabled={
    canBuilder
      ? undefined
      : { reason: "This expression isn't supported by the Builder — edit as Code." }
  }
/>
```

All other behavior is unchanged — Apply, Clear, Back, the widget-id reset effect, stepInvalid, etc.

## 9. Risks

1. **Lezer node names.** The design assumes specific node names (`AggregateExpr`, `FunctionCall`, `VectorSelector`, `MatrixSelector`, `LabelMatchers`, `LabelMatcher`, `MetricIdentifier`, `StringLiteral`, `MatchOp`, `AggregateOp`, `AggregateModifier`, `GroupingLabels`, `Duration`, `FunctionIdentifier`). If the installed grammar uses different names, walker constants adjust. Task 1 of the plan dumps the AST for three known inputs and confirms.
2. **Semantic round-trip only.** `metric{job='api'}` → parse → regenerate → `metric{job="api"}`. String differs; semantics don't. We accept this; tests assert only semantic round-trip via `buildExpression` equality.
3. **`{filter}`-only shapes.** Valid PromQL, builder doesn't support — parser returns `null`, Builder tab stays disabled. Users who want this stay in Code mode.
4. **Safeguard effect looping.** The `useEffect` that flips `mode` back to Code when `canBuilder` becomes false writes state during render resolution. React batches state updates, so no infinite loop — but tests exercise the edge case (start in Builder with parseable expr; swap widget to one with unparseable expr; assert mode resets to code).
5. **Performance on keystroke.** `parseExpression` runs inside `useMemo([expr])`, so it's fine while typing single characters. Even 10k-char PromQL parses in <1 ms with Lezer. Not debounced.
6. **Backtick-quoted label values.** PromQL permits `` `raw` `` strings. Our builder never emits them; `unquote` leaves them unchanged, which would produce a wrong value string. Acceptable — the user is writing non-builder shape and the parser will likely fail elsewhere (e.g., if they used backticks because they needed raw escaping we don't handle). Edge case noted.
7. **Grammar upgrades.** `@prometheus-io/lezer-promql` is pinned in `pnpm-lock.yaml`. Future upgrades require re-running Task 1's AST-dump verification.

## 10. Open questions (deferred)

- **Partial parse with warnings** — populate the builder from the parseable subtree of `sum(up) / 2` and warn about the dropped suffix. Clearer error UX, but harder to get right. Deferred.
- **Backtick strings** — if users frequently write PromQL with backticks, extend `unquote` to handle them. Not a common pattern in the dev dataset.
- **Debounced parsing** — only needed if keystroke performance becomes a visible issue.
- **Typed AST helpers from `@prometheus-io/codemirror-promql`** — that package may expose a typed walker that's friendlier than raw Lezer. Not needed for MVP; raw Lezer suffices.
