# P3.2 — Per-widget PromQL config + editor UI Design

**Status:** Design approved 2026-04-29 — ready for implementation plan.
**Depends on:** P3.1 (Prometheus wiring: proxy routes, `useQueryRange`, `buildLineOption`).
**Estimate:** ~20 engineering hours (~2.5 days).

Phase 2 of 3 for the Prometheus integration. P3.1 proved the end-to-end pipeline on a standalone demo page. P3.2 integrates it into real widgets for the 5 series kinds (line, area, bar, stacked-bar, scatter) with a per-widget query editor. P3.3 covers non-series kinds (KPI, gauge, table, pie/donut).

---

## 1. Goal

Let a user select a series widget in edit mode and attach a PromQL expression to it. The widget then renders live Prometheus data via the existing `<EchartsWidget>`. Save persists the query through the existing PUT path. Without a query, the widget falls back to its existing sample or backend-data behavior.

## 2. Scope

**In scope**
- Extend `WidgetRef` schema with optional `query: { expr, step? }`.
- New `WidgetAdapter.seriesType?: "line" | "bar" | "area" | "stacked-bar" | "scatter"` field.
- Populate `seriesType` on the 5 series adapters.
- `buildSeriesOption(resp, seriesType)` — generalizes P3.1's `buildLineOption`.
- `<SeriesChart>` component — shared renderer that uses `useQueryRange` + `buildSeriesOption` when a query is present.
- `<QueryEditor>` component — textarea for `expr`, optional `step` input, Apply + Clear + Back buttons.
- Selection state on the edit page (`selectedWidgetId`). Click a widget → select. Click canvas background → deselect.
- Right-side drawer swaps mode: palette when nothing selected, query editor when a widget is selected.
- Canvas `WidgetByKind` in both `DashboardCanvas` and `EditableDashboardCanvas` routes queried series widgets through `<SeriesChart>`; extracted to a shared helper.
- `stableKey` extended so query changes flip `isDirty`.
- `useQueryRange` gains an `enabled: !!expr` guard.
- Save-success merge: on successful save, edit-page state merges the POSTed widgets (which carry `query`) over the re-fetched fixture widgets (which don't), so the user's in-session queries don't get clobbered.
- Unit tests for builder, editor, schema. E2E extension for the add-query + save flow.

**Out of scope** (deferred)
- Per-widget time range picker (start/end/range) — range is baked to "last 1h" via `useQueryRange` defaults.
- Label / metric-name autocomplete.
- Query variables, dashboard-level time range.
- The other 14 widget kinds (kpi, gauge, table, pie, donut, funnel, tree, sunburst, treemap, heatmap, sankey, histogram, box-plot, candlestick).
- Persisting queries cross-reload against WireMock (the `/widgets/w-cpu-*/data.xml` fixtures are static; queries set in edit mode don't round-trip through them).
- Query multi-series aliasing / legend override.

## 4. New components

### 4.1 `src/widgets/promql/series-option.ts`

Generalize `buildLineOption` into a type-parameterized builder:
```ts
import type { EChartsCoreOption } from "@/widgets/echarts-core";
import type { PromRangeResponse, PromMatrixResult } from "@/server/schemas/prometheus";
import type { SeriesType } from "@/widgets/adapter";

function labelFor(metric: Record<string, string>): string {
  const { __name__, ...rest } = metric;
  const tail = Object.entries(rest).map(([k, v]) => `${k}="${v}"`).join(", ");
  if (__name__) return tail ? `${__name__}{${tail}}` : __name__;
  return tail || "series";
}

function parseValue(v: string): number | null {
  const n = parseFloat(v);
  return Number.isFinite(n) ? n : null;
}

function baseSeries(s: PromMatrixResult, seriesType: SeriesType) {
  const name = labelFor(s.metric);
  const data = s.values.map(([t, v]) => [t * 1000, parseValue(v)] as const);
  switch (seriesType) {
    case "line":        return { name, type: "line",    smooth: true, showSymbol: false, connectNulls: false, data };
    case "area":        return { name, type: "line",    smooth: true, showSymbol: false, connectNulls: false, areaStyle: {}, data };
    case "bar":         return { name, type: "bar",     data };
    case "stacked-bar": return { name, type: "bar",     stack: "total", data };
    case "scatter":     return { name, type: "scatter", symbolSize: 6, data };
  }
}

export function buildSeriesOption(resp: PromRangeResponse, seriesType: SeriesType): EChartsCoreOption {
  const series = resp.data.result.map((s) => baseSeries(s, seriesType));
  return {
    tooltip: { trigger: "axis" },
    legend: { data: series.map((s) => s.name), textStyle: { color: "#cbd5e1" } },
    grid: { left: 40, right: 20, top: 40, bottom: 30 },
    xAxis: {
      type: "time",
      axisLine: { lineStyle: { color: "#475569" } },
      axisLabel: { color: "#94a3b8" },
    },
    yAxis: {
      type: "value",
      axisLine: { lineStyle: { color: "#475569" } },
      axisLabel: { color: "#94a3b8" },
      splitLine: { lineStyle: { color: "#1e293b" } },
    },
    series,
  };
}
```

P3.1's `buildLineOption` is kept as a one-liner for the demo page:
```ts
// src/widgets/promql/line-option.ts (rewritten)
import { buildSeriesOption } from "./series-option";
import type { PromRangeResponse } from "@/server/schemas/prometheus";

export function buildLineOption(resp: PromRangeResponse) {
  return buildSeriesOption(resp, "line");
}
```

Existing `tests/unit/line-option.test.ts` continues to pass — the delegation preserves behavior.

### 4.2 `src/components/widgets/SeriesChart.tsx`

Shared renderer for the 5 series kinds when a query is present:
```tsx
"use client";

import type { WidgetRef } from "@/server/schemas/widget";
import type { SeriesType } from "@/widgets/adapter";
import { useQueryRange } from "@/hooks/useQueryRange";
import { EchartsWidget } from "@/components/widgets/EchartsWidget";
import { buildSeriesOption } from "@/widgets/promql/series-option";

interface Props {
  widget: WidgetRef;
  seriesType: SeriesType;
}

export function SeriesChart({ widget, seriesType }: Props) {
  const expr = widget.query?.expr ?? "";
  const step = widget.query?.step;
  const { data, isLoading, error } = useQueryRange(expr, { step });

  if (!expr) {
    return (
      <div className="text-muted-foreground text-sm">
        No query — select this widget and add a PromQL expression.
      </div>
    );
  }
  if (isLoading) return <div className="text-muted-foreground">loading…</div>;
  if (error) return <div className="text-red-400 text-sm">Query failed: {error.message}</div>;
  if (!data || data.data.result.length === 0) {
    return <div className="text-muted-foreground text-sm">No samples.</div>;
  }
  return <EchartsWidget option={buildSeriesOption(data, seriesType)} />;
}
```

Note the function checks `!expr` at the top; combined with `useQueryRange`'s new `enabled` guard (§4.3), no fetch fires for empty expr.

### 4.3 `src/hooks/useQueryRange.ts` — add `enabled` guard

Update the hook:
```ts
export function useQueryRange(expr: string, opts: QueryRangeOptions = {}) {
  return useQuery({
    queryKey: ["promql-range", expr, opts.start, opts.end, opts.step],
    queryFn: () => fetchRange(expr, opts),
    refetchInterval: opts.refetchIntervalMs ?? 15_000,
    enabled: expr.trim().length > 0,
  });
}
```

The `<SeriesChart>` short-circuits visually on empty expr; the hook's `enabled` guard is defense in depth so no fetch fires at all.

### 4.4 `src/components/widgets/QueryEditor.tsx`

```tsx
"use client";

import { useState } from "react";
import { Button } from "@/components/ui/button";
import type { WidgetRef } from "@/server/schemas/widget";
import type { WidgetQuery } from "@/server/schemas/widget";

interface Props {
  widget: WidgetRef;
  onApply: (next: WidgetQuery | undefined) => void;
  onBack: () => void;
}

export function QueryEditor({ widget, onApply, onBack }: Props) {
  const [expr, setExpr] = useState(widget.query?.expr ?? "");
  const [step, setStep] = useState<string>(
    widget.query?.step !== undefined ? String(widget.query.step) : "",
  );

  const trimmed = expr.trim();
  const parsedStep = step === "" ? undefined : Number(step);
  const stepInvalid = step !== "" && (!Number.isFinite(parsedStep) || (parsedStep as number) <= 0);

  const current = widget.query;
  const nextIsSame =
    current !== undefined &&
    trimmed === current.expr &&
    parsedStep === current.step;
  const applyDisabled = trimmed.length === 0 || stepInvalid || nextIsSame;

  const handleApply = () => {
    onApply({ expr: trimmed, ...(parsedStep !== undefined ? { step: parsedStep } : {}) });
  };
  const handleClear = () => {
    setExpr("");
    setStep("");
    onApply(undefined);
  };

  return (
    <aside className="w-64 overflow-y-auto border-l border-border bg-card p-4">
      <button
        type="button"
        onClick={onBack}
        className="mb-4 text-xs text-muted-foreground hover:text-foreground"
      >
        ← Widgets
      </button>
      <div className="mb-3 text-sm">
        <span className="text-muted-foreground">Editing: </span>
        <span className="font-medium">{widget.title}</span>
      </div>
      <label className="mb-2 block text-xs uppercase tracking-wide text-muted-foreground">
        PromQL expression
      </label>
      <textarea
        aria-label="PromQL expression"
        value={expr}
        onChange={(e) => setExpr(e.target.value)}
        rows={5}
        className="mb-3 w-full rounded border border-border bg-background p-2 font-mono text-xs"
      />
      <label className="mb-1 block text-xs uppercase tracking-wide text-muted-foreground">
        Step (seconds, optional)
      </label>
      <input
        aria-label="Step seconds"
        type="number"
        min={1}
        value={step}
        onChange={(e) => setStep(e.target.value)}
        className="mb-3 w-full rounded border border-border bg-background p-2 text-xs"
      />
      {stepInvalid && (
        <div className="mb-2 text-xs text-red-400">Step must be a positive number.</div>
      )}
      <div className="flex gap-2">
        <Button size="sm" onClick={handleApply} disabled={applyDisabled}>Apply</Button>
        <Button size="sm" variant="ghost" onClick={handleClear} disabled={!current && trimmed.length === 0}>
          Clear query
        </Button>
      </div>
    </aside>
  );
}
```

## 5. Canvas routing + selection

### 5.1 Shared `WidgetByKind` helper

Extract the kind-routing logic to a new file so both canvases share it:

```tsx
// src/widgets/render.tsx
import type { WidgetRef } from "@/server/schemas/widget";
import { WIDGET_ADAPTERS } from "@/widgets/adapter";
import { EchartsWidget } from "@/components/widgets/EchartsWidget";
import { SeriesChart } from "@/components/widgets/SeriesChart";

export function WidgetByKind({ widget }: { widget: WidgetRef }) {
  const adapter = WIDGET_ADAPTERS[widget.kind];
  if (adapter.seriesType && widget.query?.expr) {
    return <SeriesChart widget={widget} seriesType={adapter.seriesType} />;
  }
  if (adapter.Renderer) return <adapter.Renderer widget={widget} />;
  if (adapter.buildOption) return <EchartsWidget option={adapter.buildOption(widget)} />;
  return <div className="text-red-400">Unsupported: {widget.kind}</div>;
}
```

Both `DashboardCanvas` and `EditableDashboardCanvas` delete their local `WidgetByKind` and import this one.

### 5.2 Selection in `EditableDashboardCanvas`

Add props:
```ts
interface Props {
  /* existing */ width, height, widgets, onChange,
  selectedId: string | null;
  onSelect: (id: string | null) => void;
}
```

- The outer `<div className="overflow-auto">` gets `onClick={() => onSelect(null)}`.
- Each widget's inner container gets `onClick={(e) => { e.stopPropagation(); onSelect(w.id); }}`.
- When `selectedId === w.id`, that container gets `ring-2 ring-primary` for visual selection feedback.
- The existing × remove button keeps `e.stopPropagation()` on click so delete doesn't also fire a select.
- When a widget is removed, if it was selected, clear selection: `onChange(next); if (selectedId === w.id) onSelect(null);`.

### 5.3 Edit page integration

### 5.4 `stableKey` extension

Update to include `query` so applying a query flips `isDirty`:
```tsx
function stableKey(widgets: WidgetRef[]): string {
  return JSON.stringify(
    widgets.map((w) => [w.id, w.kind, w.title, w.x, w.y, w.w, w.h, w.query ?? null]),
  );
}
```

`w.query ?? null` keeps undefined and null equivalent, avoiding false dirty flags.

### 5.5 Save-success merge (critical)

The existing `useSaveDashboard.onSuccess` does `qc.setQueryData(["dashboard", id], saved)`. `saved` comes from the route's re-GET of the fixture — which doesn't carry `<query>` XML elements. So the `saved.widgets` array won't have the queries the user just applied.

Fix, in the edit page's `handleSave`:

```tsx
const handleSave = () => {
  if (editWidgets === null || editName === null) return;
  const snapshotWidgets = editWidgets;   // capture at send time
  save.mutate(
    { widgets: snapshotWidgets, name: editName },
    {
      onSuccess: (saved) => {
        // WireMock/Seagull may return widgets without the query field we just
        // sent (fixture is static). Reattach the queries from what we just
        // POSTed so the in-session view is consistent.
        const byId = new Map(snapshotWidgets.map((w) => [w.id, w]));
        const mergedWidgets = saved.widgets.map((w) => {
          const source = byId.get(w.id);
          return source?.query ? { ...w, query: source.query } : w;
        });
        // Write through the cache so useDashboard readers see the merged view.
        qc.setQueryData(["dashboard", id], { ...saved, widgets: mergedWidgets });
        router.push(viewHref);
      },
    },
  );
};
```

This requires pulling `useQueryClient` into the edit page:
```tsx
import { useQueryClient } from "@tanstack/react-query";
const qc = useQueryClient();
```

**Caveat**: this is a dev-only crutch to work around the WireMock fixture being static. On a real OpMon backend that persists the query field (or a WireMock scenario that echoes it), the merge becomes a no-op — the saved widgets already carry queries. Documented in the existing `saveDashboard` TODO as an extension of the same staging-validation concern.

## 7. Error handling summary

| Failure | Observed | Handled by |
|---|---|---|
| `expr === ""` | `<SeriesChart>` renders "No query — …" placeholder | SeriesChart short-circuit |
| `expr` is whitespace-only | `useQueryRange.enabled: false` → no fetch; SeriesChart renders placeholder | Hook guard + short-circuit |
| Prometheus proxy returns 502 PromQL error | React Query sets `error` → SeriesChart renders red "Query failed: …" | SeriesChart error branch |
| Proxy returns 502 upstream | Same as above, message includes "upstream" tag | Same branch |
| Empty result set (`data.result.length === 0`) | "No samples." placeholder | SeriesChart empty branch |
| Save returns 409 (duplicate name) | Existing red banner at top of edit page | P2.4 infrastructure |
| Applied query lost on round-trip | Save-success merge reattaches query from in-memory snapshot | edit page `handleSave` |

## 8. Risks

1. **Seagull silently drops nested `query` field.** Extension of the existing `widgets`-array TODO. The save-success merge (§5.5) mitigates the in-session UX. Real staging validation still needed before promoting.
2. **Save-success merge is dev-only crutch.** Remove (or make conditional) when a persisting backend lands. Called out in both the hook site and the TODO.
3. **Polling load.** Every queried series widget hits `/api/promql/query_range` every 15s. A dashboard with 10 queried widgets = 10 concurrent calls every 15s. Fine at current scale; could virtualize by visibility later.
4. **Click-to-select vs. drag.** react-rnd emits a click after a drag. If we end up selecting every time the user drags, we'll need to distinguish on `mousedown` vs `mouseup` position delta. Integration risk — worth a quick manual check during implementation.
5. **Selection stale after delete.** If the selected widget is removed from the palette's × button, the editor would show stale state. Fix: clear `selectedWidgetId` when its widget is removed.
6. **No range picker**. "Last 1h" is hard-coded. Users who want "last 24h" can't get it in P3.2. Explicitly deferred.
7. **Textarea caret vs. drag**. The query textarea is inside the drawer, not on the canvas, so there's no react-rnd conflict. Still, worth sanity-checking that typing in the textarea doesn't trigger any canvas interaction.

## 9. Open questions (deferred)

- **Auth** — still no Prometheus auth. P3.3 or later.
- **Query variables / dashboard-level time range** — Grafana concept; deferred.
- **Label/metric-name autocomplete** — Prometheus exposes `/api/v1/labels` etc.; could back an autocomplete layer in a future phase.
- **Non-series kinds** — all 15 other adapters need specialized mapping. P3.3.
- **Live-edit preview** — currently the user must click Apply to trigger a fetch. A debounce-on-type could auto-preview. Deferred — Apply is explicit, matches Grafana's "Run queries" button.
