# P3.4a ‚Äî CodeMirror PromQL editor + autocomplete Design

**Status:** Design approved 2026-05-02 ‚Äî ready for implementation plan.
**Depends on:** P3.2 (QueryEditor), P3.3 (full PromQL kind coverage). Reuses P3.1's `callPrometheus` client.
**Estimate:** ~18 engineering hours (~2.25 days).

This is **phase 4a of the Prometheus integration**. P3.4b (visual query builder) is deferred until after shipping.

---

## 1. Goal

Replace the plain `<textarea>` in the widget QueryEditor with a CodeMirror 6 editor that provides PromQL syntax highlighting, autocomplete (metric names, label names, label values, functions), and lint diagnostics ‚Äî driven by `@prometheus-io/codemirror-promql`, the same extension Prometheus and Grafana use.

## 2. Scope

**In scope**
- `<PromQLEditor>` client component built on CodeMirror 6 with the `@prometheus-io/codemirror-promql` extension.
- Lazy-loaded via `next/dynamic({ssr:false})` so the ~500 KB of editor deps don't hit the main bundle.
- Dark theme (`@codemirror/theme-one-dark` plus small overrides to match app tokens).
- Keyboard shortcut: **Cmd/Ctrl+Enter** triggers Apply while focused. Ctrl+Space (library default) invokes autocomplete manually. Autocomplete also triggers automatically while typing.
- Three new GET proxy routes for the library to discover metadata:
  - `GET /api/promql/labels` ‚Äî proxies `GET /api/v1/labels`.
  - `GET /api/promql/label/[name]/values` ‚Äî proxies `GET /api/v1/label/<name>/values`.
  - `GET /api/promql/metadata` ‚Äî proxies `GET /api/v1/metadata`.
- Three new Zod response schemas appended to `src/server/schemas/prometheus.ts`.
- Custom `PrometheusClient` implementation at `src/widgets/promql/prom-client.ts` that satisfies the library's interface by calling our proxy routes.
- `<QueryEditor>` swaps its textarea for `<PromQLEditor>`. The `aria-label="PromQL expression"` is preserved via `EditorView.contentAttributes` so the existing Playwright selector keeps working.
- Unit tests for the three new routes and the custom `PrometheusClient`.
- Updated unit tests for `QueryEditor` (mock `PromQLEditor` as a textarea stand-in so the existing state-derivation assertions still run).

**Out of scope** (deferred)
- Visual query builder (P3.4b).
- `GET /api/v1/series` matcher-refined label/value discovery. The library supports scoped lookups; for MVP, autocomplete returns the global label set regardless of the current expression's metric name.
- Query variables, dashboard-level time range, recording/alerting rules.
- Code‚Üíbuilder bidirectional parsing (that's P3.4b's problem ‚Äî if/when we build the visual builder).
- Per-kind hiding of the `step` field in the editor.
- Unit formatters, Grafana-style "reducer" dropdowns, gauge thresholds ‚Äî all carried over as deferred from P3.3.

## 5. Server-side routes

### 5.1 Schemas ‚Äî append to `src/server/schemas/prometheus.ts`:

```ts
export const PromLabelsResponseSchema = z.object({
  status: z.literal("success"),
  data: z.array(z.string()),
});
export type PromLabelsResponse = z.infer<typeof PromLabelsResponseSchema>;

export const PromLabelValuesResponseSchema = z.object({
  status: z.literal("success"),
  data: z.array(z.string()),
});
export type PromLabelValuesResponse = z.infer<typeof PromLabelValuesResponseSchema>;

export const PromMetadataResponseSchema = z.object({
  status: z.literal("success"),
  data: z.record(
    z.array(
      z.object({
        type: z.string(),
        help: z.string(),
        unit: z.string(),
      }),
    ),
  ),
});
export type PromMetadataResponse = z.infer<typeof PromMetadataResponseSchema>;
```

### 5.2 Route: `app/api/promql/labels/route.ts`

```ts
import { NextResponse } from "next/server";
import { callPrometheus, PrometheusError } from "@/server/prometheus/client";
import {
  PromLabelsResponseSchema,
  PromErrorResponseSchema,
} from "@/server/schemas/prometheus";

export async function GET() {
  try {
    const raw = await callPrometheus("/api/v1/labels", new URLSearchParams());
    const errCheck = PromErrorResponseSchema.safeParse(raw);
    if (errCheck.success) {
      return NextResponse.json(
        { error: "promql", errorType: errCheck.data.errorType, message: errCheck.data.error },
        { status: 502 },
      );
    }
    const parsed = PromLabelsResponseSchema.parse(raw);
    return NextResponse.json(parsed, { headers: { "Cache-Control": "no-store" } });
  } catch (err) {
    if (err instanceof PrometheusError) {
      return NextResponse.json(
        { error: "upstream", status: err.status, message: err.message },
        { status: 502 },
      );
    }
    return NextResponse.json(
      { error: "internal", message: err instanceof Error ? err.message : String(err) },
      { status: 500 },
    );
  }
}
```

### 5.3 Route: `app/api/promql/label/[name]/values/route.ts`

```ts
import { NextResponse } from "next/server";
import { callPrometheus, PrometheusError } from "@/server/prometheus/client";
import {
  PromLabelValuesResponseSchema,
  PromErrorResponseSchema,
} from "@/server/schemas/prometheus";

export async function GET(
  _req: Request,
  ctx: { params: Promise<{ name: string }> },

  const { name } = await ctx.params;
  try {
    const raw = await callPrometheus(
      `/api/v1/label/${encodeURIComponent(name)}/values`,

    return NextResponse.json(parsed, { headers: { "Cache-Control": "no-store" } });
  } catch (err) {
    if (err instanceof PrometheusError) {
      return NextResponse.json(
        { error: "upstream", status: err.status, message: err.message },
        { status: 502 },
      );
    }
    return NextResponse.json(
      { error: "internal", message: err instanceof Error ? err.message : String(err) },
      { status: 500 },
    );
  }
}
```

## 6. Custom `PrometheusClient`

`src/widgets/promql/prom-client.ts`:

```ts
"use client";

import type {
  PrometheusClient,
} from "@prometheus-io/codemirror-promql";

async function getJson<T>(path: string): Promise<T> {
  const r = await fetch(path, { credentials: "include" });
  if (!r.ok) throw new Error(`${path} ‚Üí ${r.status}`);
  return (await r.json()) as T;
}

export function createPromClient(): PrometheusClient {
  return {
    async labelNames(): Promise<string[]> {
      const { data } = await getJson<{ data: string[] }>("/api/promql/labels");
      return data;
    },
    async labelValues(labelName: string): Promise<string[]> {
      const { data } = await getJson<{ data: string[] }>(
        `/api/promql/label/${encodeURIComponent(labelName)}/values`,
      );
      return data;
    },
    async metricNames(): Promise<string[]> {
      // Convention: metric names live under the `__name__` label.
      const { data } = await getJson<{ data: string[] }>(
        `/api/promql/label/${encodeURIComponent("__name__")}/values`,
      );
      return data;
    },
    async metricMetadata(): Promise<Record<string, Array<{ type: string; help: string; unit: string }>>> {
      const { data } = await getJson<{ data: Record<string, Array<{ type: string; help: string; unit: string }>> }>(
        "/api/promql/metadata",
      );
      return data;
    },
    // series() is used for scoped label-value lookups; stub to empty for MVP.
    async series(): Promise<Array<Record<string, string>>> {
      return [];
    },
    // Allow the library's prefix search to work even with stubs.
    async flags(): Promise<Record<string, string>> {
      return {};
    },
  };
}
```

Note: the **precise** shape of `PrometheusClient` comes from the installed library version. The implementer verifies the interface at install time and adjusts the stub methods accordingly; the shape above reflects the typical API surface but minor method name changes are expected.

### 7.2 `src/components/widgets/PromQLEditorInner.tsx` ‚Äî CodeMirror setup

```tsx
"use client";

import { useEffect, useRef } from "react";
import { EditorState } from "@codemirror/state";
import { EditorView, keymap } from "@codemirror/view";
import { basicSetup } from "codemirror";
import { oneDark } from "@codemirror/theme-one-dark";
import { PromQLExtension } from "@prometheus-io/codemirror-promql";
import { createPromClient } from "@/widgets/promql/prom-client";

interface Props {
  value: string;
  onChange: (next: string) => void;
  onApply?: () => void;
}

export default function PromQLEditorInner({ value, onChange, onApply }: Props) {
  const hostRef = useRef<HTMLDivElement>(null);
  const viewRef = useRef<EditorView | null>(null);

  // Keep latest callbacks in refs so the editor closure doesn't go stale.
  const onChangeRef = useRef(onChange);
  const onApplyRef = useRef(onApply);
  useEffect(() => {
    onChangeRef.current = onChange;
  }, [onChange]);
  useEffect(() => {
    onApplyRef.current = onApply;
  }, [onApply]);

  // Mount once on first render.
  useEffect(() => {
    if (!hostRef.current) return;

    const promql = new PromQLExtension().setComplete({
      remote: { client: createPromClient() },
    });

    const applyKey = keymap.of([
      {
        key: "Mod-Enter",
        run: () => {
          onApplyRef.current?.();
          return true;
        },
      },
    ]);

    const state = EditorState.create({
      doc: value,
      extensions: [
        basicSetup,
        oneDark,
        promql.asExtension(),
        applyKey,
        EditorView.contentAttributes.of({ "aria-label": "PromQL expression" }),
        EditorView.updateListener.of((u) => {
          if (u.docChanged) {
            onChangeRef.current(u.state.doc.toString());
          }
        }),
      ],
    });

    const view = new EditorView({ state, parent: hostRef.current });
    viewRef.current = view;
    return () => view.destroy();
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, []);

  // Sync external value changes (e.g., parent swapping widgets ‚Üí different
  // expr). Avoid feedback loops by checking equality first.
  useEffect(() => {
    const view = viewRef.current;
    if (!view) return;
    const current = view.state.doc.toString();
    if (current !== value) {
      view.dispatch({
        changes: { from: 0, to: current.length, insert: value },
      });
    }
  }, [value]);

  return (
    <div
      ref={hostRef}
      className="overflow-hidden rounded border border-border bg-background"
      style={{ minHeight: 96 }}
    />
  );
}
```

The `PromQLExtension().setComplete({remote: {client}}).asExtension()` chain reflects the library's API as of mid-2025. If the installed version expects `.setComplete({client})` without the `remote` wrapper, adapt.

## 10. Risks

1. **Bundle weight.** CodeMirror + PromQL extension is ~500 KB gzipped. Lazy-loaded via `next/dynamic({ssr:false})` so only users opening edit mode pay the cost. Brief "loading editor‚Ä¶" placeholder on first render.
2. **Library API drift.** `@prometheus-io/codemirror-promql` has refactored across versions. The implementer verifies the exact API of the installed pin (notably `PromQLExtension`'s completion configuration and the `PrometheusClient` method signatures) and adapts ¬ß6 + ¬ß7.2 accordingly. Behavior is the contract.
3. **happy-dom doesn't simulate CodeMirror.** Real-editor unit tests are impractical; we mock the `PromQLEditor` in `QueryEditor.test.tsx`. Coverage loss acceptable (library is third-party + well-tested); E2E covers the integration path.
4. **`aria-label` must land on the contenteditable div.** If missing, `getByRole("textbox", { name: "PromQL expression" })` won't match and every existing E2E spec breaks. The plan's editor task explicitly verifies this via the full-stack smoke step that runs the edit E2E.
5. **Metadata endpoint size.** `/api/v1/metadata` on a large Prom can return many kilobytes. Our docker self-scrape has a small metadata set. If this becomes an issue at scale, the library supports debouncing; we rely on its defaults.
6. **Autocomplete network failures.** If a metadata or labels fetch 500s, CodeMirror silently returns empty completions. Acceptable. No user-facing error surface.
7. **Cmd+Enter gotcha on Mac vs Windows.** CodeMirror's `Mod-` alias maps to Cmd on Mac / Ctrl on Windows automatically. No branching needed.
8. **The dark theme's default colors vs. our Tailwind tokens.** `oneDark` looks close but not identical to the app. The design accepts visual imperfection; a 1-hour polish pass matches key tokens (border color, background). If the user later objects to the exact palette, more adjustment is a quick follow-up.
