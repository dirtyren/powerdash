# P2.4 — Dashboard Rename (in edit mode) Design

**Status:** Design approved 2026-04-29 — ready for implementation plan.
**Depends on:** P2.2 (edit-mode scaffolding + `EditToolbar.onTitleChange`)
**Estimate:** ~3.5 engineering hours

---

## 1. Goal

Let a user rename an existing dashboard by changing its name in the existing edit-mode toolbar. Save commits both the name and layout in one PUT. No new routes, no new UI affordances.

## 2. Scope

**In scope**
- Inline-editable dashboard name on `/dashboards/[id]/edit` via the existing `EditToolbar.onTitleChange` prop.
- Extend `PutBodySchema` to accept an optional `name` field; forward it to `saveDashboard`.
- Extend `useSaveDashboard` to take `{ widgets, name? }` instead of just `widgets`.
- Extend `isDirty` on the edit page to also flip on name changes.
- Unit-test coverage for the PUT-with-name path.
- E2E update to rename during edit.

**Out of scope** (deferred)
- Delete — requires validating an unknown Seagull endpoint.
- Duplicate — client-side feature; could be a later one-task plan.
- Rename from the detail page (no separate dialog or inline detail-page edit).
- Rename from the list page.
- Changing the dashboard owner, width, or height.

## 3. Data / API changes

### 3.1 `PutBodySchema`

Current (`app/api/dashboards/[id]/route.ts`):
```ts
const PutBodySchema = z.object({
  widgets: z.array(WidgetRefSchema),
});
```

New:
```ts
const PutBodySchema = z.object({
  widgets: z.array(WidgetRefSchema),
  name: z.string().trim().min(1).optional(),
});
```

`name` is optional for back-compat with callers that only send widgets (none exist today, but the optional shape keeps the contract forgiving). Validation mirrors `DashboardSummarySchema.name` (`.trim().min(1)` — established in P2.2's final review to reject whitespace-only names at the schema boundary).

### 3.2 PUT route merge

Current:
```ts
const saved = await saveDashboard({
  ...existing,
  widgets: parsedBody.widgets,
});
```

New:
```ts
const saved = await saveDashboard({
  ...existing,
  widgets: parsedBody.widgets,
  ...(parsedBody.name !== undefined ? { name: parsedBody.name } : {}),
});
```

### 3.3 `saveDashboard` module (`src/server/seagull/dashboards.ts`)

**Unchanged.** Already accepts the full dashboard shape.

### 3.4 `useSaveDashboard` hook

Current signature:
```ts
useSaveDashboard(id).mutate(widgets: WidgetRef[])
```

New:
```ts
useSaveDashboard(id).mutate({ widgets, name }: { widgets: WidgetRef[]; name?: string })
```

The `mutationFn` forwards the object as the request body. Cache invalidation behavior (`qc.setQueryData(["dashboard", id], saved)` + `qc.invalidateQueries({ queryKey: ["dashboards"] })`) is unchanged.

### 3.5 WireMock

**Unchanged.** The existing save mapping matches any form-encoded `json=*` body, so it doesn't care whether the envelope includes a new name or not. The post-save re-GET returns the static `get-dashboard-1.xml` fixture regardless.

## 4. Edit page wiring (`app/dashboards/[id]/edit/page.tsx`)

### 4.1 Name state

Add alongside the existing `editWidgets` state:
```ts
const [editName, setEditName] = useState<string | null>(null);
```

Initialize in the same `useEffect` that initializes `editWidgets`:
```ts
useEffect(() => {
  if (data && editWidgets === null) {
    setEditWidgets(data.widgets);
    setEditName(data.name);
  }
}, [data, editWidgets]);
```

### 4.2 Expanded `isDirty`

```ts
const isDirty = useMemo(() => {
  if (!data || editWidgets === null || editName === null) return false;
  return (
    editName !== data.name ||
    stableKey(editWidgets) !== stableKey(data.widgets)
  );
}, [data, editWidgets, editName]);
```

### 4.3 Toolbar wiring

```tsx
<EditToolbar
  title={editName ?? data.name}
  isDirty={isDirty}
  isSaving={save.isPending}
  onSave={handleSave}
  onCancel={handleCancel}
  onTitleChange={setEditName}
/>
```

The `<EditToolbar>` already supports `onTitleChange` (P2.2 Task 7) — it renders the title as an `<input aria-label="Dashboard name" />` when the callback is provided.

### 4.4 Save handler

```ts
const handleSave = () => {
  if (editWidgets === null || editName === null) return;
  save.mutate(
    { widgets: editWidgets, name: editName },
    { onSuccess: () => router.push(viewHref) },
  );
};
```

### 4.5 Cancel handler

Unchanged. The existing confirm-on-dirty covers name changes through the expanded `isDirty`.

## 5. Error handling

All error paths already exist:
- `output: -2` → `SaveDashboardError(code: -2)` → route returns 409 → hook's mutation error → `save.error` banner at the top of the edit page ("A dashboard with this name already exists.").
- Whitespace-only name → Zod parse fails at the route boundary → 422 "invalid_body".
- `output: -3 / -4 / -1`, `SeagullError`, unknown error — all already wired.

No new error surfaces or UI.

### 6.2 Unit — no `useSaveDashboard` test

Matches the existing pattern (the hook had no unit test before; it's a thin wrapper covered end-to-end via E2E).

### 6.3 E2E — extend `tests/e2e/dashboard-edit.spec.ts`

Add one rename step before the existing drag step:
```ts

await nameInput.fill("Renamed by E2E");
await expect(page.getByText(/unsaved changes/)).toBeVisible();
```

Continue with the existing drag + save. After save, we can NOT assert the new name shows on `/dashboards/1` because WireMock returns the static `get-dashboard-1.xml` fixture. Same limitation as today. The new assertion is just that the name input accepts input and dirty flips.

## 8. Risks

1. **WireMock re-GET is static.** The E2E can't assert post-save persistence for the new name. Acceptable — same trade-off as today's drag-and-save. A future WireMock scenario state machine could fix both.
2. **Duplicate-name error UX.** When a user renames to an already-taken name, the save fails with a 409 and the error banner shows. The edit state is NOT reverted — the user keeps the name in the input. They can adjust and retry. That's the desired UX; no change needed.
3. **Hook signature change is a breaking API.** Only one caller (`/[id]/edit/page.tsx`) — updated atomically in the plan. No other consumers.
