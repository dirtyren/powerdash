# PowerDash pivot bundle Design

**Status:** Design 2026-05-03 — ready for implementation plan.
**Depends on:** None. Supersedes the stateful mock-api (commit `0d61ee1`), which is deleted in this change.
**Estimate:** ~20–26 engineering hours (~3 days).

A four-change bundle that pivots the project from "OpMon dashboard migration" to "PowerDash: a Grafana-like Prometheus dashboard tool":

1. **Drop OpMon/Seagull entirely.** Persist dashboards in Postgres via Drizzle ORM. Delete `src/server/seagull/*`, the widget-data XML route, the legacy non-Prometheus widgets, and the mock-api container.
2. **Rename Davinci → PowerDash** across code, package, README, env vars.
3. **Sidebar dashboard list + back-to-home nav** on edit/create pages.
4. **Replace the plain `<select>` metric picker** with a searchable input + filtered listbox in the QueryBuilder.

The bundle is one scope because changes 1–4 touch overlapping files; splitting would cause double-edits.

---

## 1. Goal

PowerDash is a standalone Prometheus dashboard tool. After this change:
- Saves survive container restarts (real Postgres, not a mock).
- The sidebar is a working navigation surface — every dashboard is one click away from anywhere.
- Creating/editing a dashboard has an obvious path back to the list.
- Picking a metric in the builder works like any modern dashboarding tool (type-filter, keyboard select).
- No code or strings reference OpMon, Davinci, Seagull, or the legacy Flex client.

Non-goals: auth, user management, folders/organizations, Prometheus-source configuration UI, dashboard sharing/export. The env vars `DATABASE_URL` and `PROMETHEUS_BASE_URL` stay the only configuration.

## 2. Scope

**In scope**
- New `src/server/db/` layer using Drizzle ORM + `node-postgres`.
- New `dashboards` Postgres table (single table, widgets in JSONB).
- Drizzle Kit migrations committed under `drizzle/`.
- `scripts/migrate.ts` + `scripts/seed.ts`. Migrations run on container startup.
- Seed inserts "Infrastructure Overview" (3 widgets) if the table is empty.
- Rewritten `app/api/dashboards/route.ts` (GET list + POST create) and `app/api/dashboards/[id]/route.ts` (GET + PUT) using the new DB layer.
- Deletion: `docker/mock-api/`, `src/server/seagull/`, `src/hooks/useWidgetData*.ts`, `app/api/widgets/[id]/data/route.ts`, legacy widget components (`KpiTile`, `LineChart`, `DataTable`) and their tests.
- Rename: `Davinci` → `PowerDash` in `app/layout.tsx`, `src/components/AppShell.tsx`, `README.md`, `package.json.name`. `opmon` → `powerdash` in `docker-compose.yml` DB user/name + env var, plus code comment sweep.
- Sidebar: renders `useDashboards()` output as links; top label becomes a `<Link href="/">`.
- Edit toolbar: "← Dashboards" link (covers both `/dashboards/[id]/edit` and `/dashboards/new`).
- New `src/components/widgets/MetricPicker.tsx` combobox. `QueryBuilder.tsx` swaps its `<select>` for `<MetricPicker>`.
- New tests: ~12 unit (DB repo + MetricPicker). Existing 5 E2E specs continue to pass — `edit-persist` becomes a real persistence check against Postgres.

**Out of scope** (deferred)
- Auth, users, permissions. `owner` field is dropped entirely; no user concept in schema.
- Folders / dashboard organization.
- Multiple Prometheus sources.
- Dashboard import/export (JSON upload/download).
- Prometheus-backed alternatives for what legacy widgets used to do (delete, don't port).
- Fuzzy metric matching (substring match only).
- Migration from the old mock-api's in-memory state. Delete outright; no backwards compat.

## 5. Postgres persistence

**Schema** (`src/server/db/schema.ts`):

```ts
import { pgTable, text, integer, jsonb, timestamp, uuid } from "drizzle-orm/pg-core";
import type { Widget } from "../schemas/widget";

export const dashboards = pgTable("dashboards", {
  id: uuid("id").primaryKey().defaultRandom(),
  name: text("name").notNull(),
  width: integer("width").notNull().default(1920),
  height: integer("height").notNull().default(1080),
  widgets: jsonb("widgets").$type<Widget[]>().notNull().default([]),
  createdAt: timestamp("created_at").notNull().defaultNow(),
  updatedAt: timestamp("updated_at").notNull().defaultNow(),
});
```

`uuid().defaultRandom()` so ids are stable and non-predictable. `widgets` is typed via Drizzle's `$type<>` — it stores raw JSON but TypeScript treats it as `Widget[]`. Zod still validates on write at the API boundary.

**Connection** (`src/server/db/client.ts`):

```ts
import { drizzle } from "drizzle-orm/node-postgres";
import { Pool } from "pg";
import * as schema from "./schema";

const globalForDb = global as unknown as { pool?: Pool };
const pool = globalForDb.pool ?? new Pool({ connectionString: process.env.DATABASE_URL });
if (process.env.NODE_ENV !== "production") globalForDb.pool = pool;

export const db = drizzle(pool, { schema });
```

Singleton-guard pattern to avoid pool leaks on Next.js hot reload.

**Repo** (`src/server/db/dashboards.ts`):

```ts
import { eq, asc } from "drizzle-orm";
import { db } from "./client";
import { dashboards } from "./schema";
import type { Dashboard, CreateDashboard, DashboardSummary } from "../schemas/dashboard";

export async function listDashboards(): Promise<DashboardSummary[]> {
  const rows = await db.select({ id: dashboards.id, name: dashboards.name })
    .from(dashboards).orderBy(asc(dashboards.name));
  return rows;
}

export async function getDashboard(id: string): Promise<Dashboard | null> {
  const [row] = await db.select().from(dashboards).where(eq(dashboards.id, id)).limit(1);
  return row ?? null;
}

export async function createDashboard(input: CreateDashboard): Promise<Dashboard> {
  const [row] = await db.insert(dashboards).values({
    name: input.name,
    width: input.width, height: input.height,
    widgets: input.widgets ?? [],
  }).returning();
  if (!row) throw new Error("insert returned no row");
  return row;
}

export async function updateDashboard(id: string, input: Partial<Dashboard>): Promise<Dashboard | null> {
  const [row] = await db.update(dashboards)
    .set({
      ...(input.name !== undefined ? { name: input.name } : {}),
      ...(input.width !== undefined ? { width: input.width } : {}),
      ...(input.height !== undefined ? { height: input.height } : {}),
      ...(input.widgets !== undefined ? { widgets: input.widgets } : {}),
      updatedAt: new Date(),
    })
    .where(eq(dashboards.id, id))
    .returning();
  return row ?? null;
}
```

**Migrations** — `drizzle.config.ts`:

```ts
import type { Config } from "drizzle-kit";
export default {
  schema: "./src/server/db/schema.ts",
  out: "./drizzle",
  dialect: "postgresql",

import { db } from "../src/server/db/client";

async function main() {
  await migrate(db, { migrationsFolder: "./drizzle" });
  console.log("migrations applied");
  process.exit(0);
}
main().catch((e) => { console.error(e); process.exit(1); });
```

**Seed** — `scripts/seed.ts`:

```ts
import { db } from "../src/server/db/client";
import { dashboards } from "../src/server/db/schema";

async function main() {
  const existing = await db.select({ id: dashboards.id }).from(dashboards).limit(1);
  if (existing.length > 0) { console.log("seed skipped (non-empty)"); return; }
  await db.insert(dashboards).values([{
    name: "Infrastructure Overview",
    width: 1920, height: 1080,
    widgets: [
      { id: "w-cpu-kpi",     kind: "kpi",   title: "CPU %",          x:  20, y:  20, w:  260, h: 160 },
      { id: "w-cpu-line",    kind: "line",  title: "CPU over time",  x: 300, y:  20, w:  720, h: 320 },
      { id: "w-hosts-table", kind: "table", title: "Hosts",          x:  20, y: 360, w: 1000, h: 320 },
    ],
  }]);
  console.log("seeded 1 dashboard");
  process.exit(0);
}
main().catch((e) => { console.error(e); process.exit(1); });
```

## 6. Schema changes

`src/server/schemas/dashboard.ts` before:

```ts
export const DashboardSummarySchema = z.object({
  id: z.string().min(1),
  name: z.string().trim().min(1),
  owner: z.string().min(1),
});
```

After — drop `owner`, keep the string id (Postgres returns UUIDs as strings):

```ts
export const DashboardSummarySchema = z.object({
  id: z.string().uuid(),
  name: z.string().trim().min(1),
});
export const DashboardSchema = DashboardSummarySchema.extend({
  width: z.coerce.number().int().positive().catch(1920),
  height: z.coerce.number().int().positive().catch(1080),
  widgets: z.array(WidgetRefSchema),
  createdAt: z.date().optional(),
  updatedAt: z.date().optional(),
});
export const CreateDashboardSchema = DashboardSchema.omit({ id: true, createdAt: true, updatedAt: true });
```

`WidgetRefSchema` (src/server/schemas/widget.ts) is unchanged.

## 7. API routes (Postgres-backed)

`app/api/dashboards/route.ts`:

```ts
export async function GET() {
  const data = await listDashboards();
  return NextResponse.json(data, { headers: { "Cache-Control": "no-store" } });
}

export async function POST(req: Request) {
  const raw = await req.json();
  const parsed = CreateDashboardSchema.safeParse(raw);
  if (!parsed.success) {
    return NextResponse.json({ error: "invalid_body", message: parsed.error.message }, { status: 422 });
  }
  const saved = await createDashboard(parsed.data);
  return NextResponse.json(saved, { status: 201, headers: { "Cache-Control": "no-store" } });
}
```

`app/api/dashboards/[id]/route.ts`:

```ts
const PutBodySchema = z.object({
  widgets: z.array(WidgetRefSchema),
  name: z.string().trim().min(1).optional(),
  width: z.coerce.number().int().positive().optional(),
  height: z.coerce.number().int().positive().optional(),
});

export async function GET(_req: Request, ctx: { params: Promise<{ id: string }> }) {
  const { id } = await ctx.params;
  const d = await getDashboard(id);
  if (!d) return NextResponse.json({ error: "not_found" }, { status: 404 });
  return NextResponse.json(d, { headers: { "Cache-Control": "no-store" } });
}

export async function PUT(req: Request, ctx: { params: Promise<{ id: string }> }) {
  const { id } = await ctx.params;
  const raw = await req.json();
  const parsed = PutBodySchema.safeParse(raw);
  if (!parsed.success) {
    return NextResponse.json({ error: "invalid_body", message: parsed.error.message }, { status: 422 });
  }
  const saved = await updateDashboard(id, parsed.data);
  if (!saved) return NextResponse.json({ error: "not_found" }, { status: 404 });
  return NextResponse.json(saved, { headers: { "Cache-Control": "no-store" } });
}
```

Error shape unchanged (`{error, message}`), so client hooks need no changes.

## 9. MetricPicker

New component at `src/components/widgets/MetricPicker.tsx`. Controlled combobox: `value: string`, `onChange: (v: string) => void`, `metrics: string[]`. Features:

- Typing filters by case-insensitive substring.
- 200-item visible cap (silent; no "more" hint).
- Arrow up/down navigates active option; Enter commits; Escape closes.
- Mouse hover sets active; click (onMouseDown) commits.
- Click outside the root element closes.
- External `value` prop is synced to the input on change (covers the parse-expression populate path).

`QueryBuilder.tsx` replaces its native `<select>` block with:

```tsx
<MetricPicker
  value={state.metric}
  onChange={(next) => onChange({ ...state, metric: next })}
  metrics={metricNames ?? []}
  placeholder="Select a metric…"
/>
```

6 unit tests (`tests/unit/MetricPicker.test.tsx`): initial render, typing filters, selecting an option, arrow-Enter flow, external value sync, click-outside closes.

## 14. Risks

1. **Running migrations on every container start.** Drizzle migrations are idempotent (skip already-applied), so safe. Still adds ~200 ms to startup. Acceptable.
2. **Test DB isolation.** Unit tests truncate `dashboards` before each case. If two test files run in parallel and both truncate, state leaks. Vitest default is single-thread per file but parallel across files — mitigate by setting `vitest`'s `pool: "forks"` + `isolate: true` for DB tests, or by adding `poolOptions.forks.singleFork: true` project-wide. The plan will pin this.
3. **UUID column vs. current client expectations.** Hook code treats `id` as string — no change. E2E specs that URL-match against `/dashboards/\d+$/` get loosened to `/dashboards/[0-9a-f-]+$/` (UUID regex).
4. **Migrations that break during rollout.** This is a greenfield DB (no production users); if migrations fail in dev, delete the Postgres volume and re-run. Acceptable for MVP.
5. **Drizzle version churn.** Pin `drizzle-orm@^0.33` / `drizzle-kit@^0.24`, `pg@^8.13`. Upgrade deliberately later.
6. **Hot-reload connection leaks.** Singleton-guard pattern in `client.ts` addresses this. Verified via visual connection count in `pg_stat_activity` during manual dev.
7. **MetricPicker performance on very large metric lists.** The 200-item visible cap prevents React from reconciling thousands of `<li>`s. If a user has 50k metrics, the filter still runs `.filter().slice()` on every keystroke — O(n) per keystroke. 50k × a few chars is well under 1 ms in practice. No memo needed.
8. **Breaking hot-reload on `drizzle.config.ts` changes.** Drizzle Kit reads the config at generate/migrate time only; dev server is unaffected.

## 15. Open questions (deferred)

- **Auth.** Will be its own change. For now, `owner` is dropped from the schema and any user concept is absent.
- **Dashboard sharing / export.** JSON import/export is a future ticket.
- **Prometheus datasource UI.** `PROMETHEUS_BASE_URL` env var stays the only knob. A settings page can come later.
- **Fuzzy metric matching.** Substring match is fine for `http_requests_total`-style names. Revisit if users request it.
- **Migration from legacy mock-api data.** Nobody was using the mock in production; delete outright.
