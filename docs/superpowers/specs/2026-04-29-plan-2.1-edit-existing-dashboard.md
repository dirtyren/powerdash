## 1. Scope

Enable users to edit the layout of an existing OpMon dashboard: drag/resize/reorder widgets on the grid, remove widgets, save the result to the seagull backend.

### In scope
- New route `/dashboards/[id]/edit`.
- Drag, resize, and reorder of existing widgets using `react-grid-layout`.
- Remove a widget from the dashboard.
- Save (persist via seagull PUT) and Cancel (discard + navigate back).
- Dirty-state tracking (Save disabled while clean).
- E2E smoke test covering the drag → save → reload-persistence cycle.

### Out of scope (reserved for Plan 2.2)
- Creating a new dashboard (`/dashboards/new`).
- Adding a widget to an existing dashboard (requires widget catalog + inline widget creation).
- Creating new widgets (metric picker, viz-type chooser).
- Rename / delete / duplicate / share-URL for dashboards.

### Out of scope (reserved for later P2 plans)
- Theming (light/dark, brand tokens).
- i18n (`next-intl`).
- Accessibility polish beyond keyboard basics included with `react-grid-layout`.
- Remaining widget types (gauge, column, pie, heatmap, status-grid, alert list, event timeline, group summary, uptime bar, SLA matrix).

The existing `src/components/DashboardGrid.tsx` (read-only) remains untouched — `EditableDashboardGrid.tsx` is a sibling for edit mode, not a replacement.

### 2.2 Component boundaries

- **`EditableDashboardGrid`** — renders widgets using `react-grid-layout`'s `<GridLayout>` in the same 12-col grid the read-only `DashboardGrid` uses. Accepts `widgets: WidgetRef[]` + `onChange(next: WidgetRef[])`. Translates between `react-grid-layout`'s `{i, x, y, w, h}` layout items and our `WidgetRef` shape. Adds a small `×` remove button per widget card (absolute-positioned overlay, visible only in edit mode).
- **`EditToolbar`** — stateless, takes `{ isDirty, isSaving, onSave, onCancel }`. Renders a sticky top bar with title, Save (primary, disabled unless dirty), Cancel (ghost).
- **`useSaveDashboard(id)`** — wraps TanStack Query `useMutation` calling `fetch('/api/dashboards/:id', { method: 'PUT', body: JSON.stringify({ widgets }) })`. On success invalidates `["dashboard", id]` and `["dashboards"]`.
- **`saveDashboard(id, widgets)` server adapter** — builds the seagull XML body with `buildDashboardXml`, calls `callSeagull` with `method: "POST"` (or `"PUT"` — confirmed in discovery), returns the parsed `Dashboard`.
- **`xml-build.ts`** — new utility exporting `buildDashboardXml(id, widgets): string`. Pure, testable, opposite of the existing `parseSeagullXml`.

### 2.3 Data flow

```
User clicks Edit on /dashboards/[id]
  → router.push('/dashboards/[id]/edit')

/dashboards/[id]/edit page:
  useDashboard(id) fetches (cache hit from view route)
  → copy widgets into local state (React useState)
  → EditableDashboardGrid renders
  → user drags/resizes/removes → setLocalWidgets(...)
  → isDirty = !deepEqual(localWidgets, originalWidgets)

User clicks Save:
  useSaveDashboard.mutate(localWidgets)
  → PUT /api/dashboards/[id] { widgets }
  → Route handler: saveDashboard(id, widgets) → buildDashboardXml → callSeagull(POST/PUT)
  → On success: invalidate query cache, router.push('/dashboards/[id]')
  → On error: toast, keep local state

User clicks Cancel:
  If isDirty: confirm("Discard changes?")
  → router.push('/dashboards/[id]')
```

### 2.4 Seagull write payload — discovery first

The exact XML shape accepted by seagull for dashboard save is not yet known in this workspace. **Task 1 of the implementation plan is a short discovery spike** (time-boxed to 1 day):

1. Read `docs/legacy/Documentacao API Integracao OpMon Dashboards.pdf`.
2. Inspect `../dashboard/flex/src/com/opservices/**/*.as` for the existing Flex client's save request (grep for `saveDashboard`, `POST`, `PUT`, `sendAndLoad`, `writeXML`).
3. Capture a real request via WireMock record/playback against a dev OpMon instance if available.
4. Document the shape as fixtures under `docker/wiremock/__files/save-dashboard-*.xml` and as notes at the top of `src/server/seagull/xml-build.ts`.

Only after discovery does the adapter implementation begin. If discovery reveals seagull uses a non-XML body (form-encoded, JSON), the adapter changes — the plan has a branch for that.

## 3. Error handling

| Failure | Behavior |
|---|---|
| PUT 502 (upstream down) | Toast: "Couldn't save. Check your connection and try again." Keep local state; Save button re-enabled. |
| PUT 409 (concurrent edit, if seagull returns it) | Toast: "Dashboard was modified elsewhere. Refresh to see the latest version." Keep local state. |
| PUT 422 (validation, e.g. invalid widget layout) | Toast with first Zod error message. Keep local state. |
| PUT 500 (internal) | Toast: "Server error while saving. Please try again." Log to Sentry (once wired in P3). |
| Zod parse fail on PUT response | Log; treat as save success (seagull persisted, our parse is stale). User is redirected to view route; the fresh fetch will re-validate. |
| Cancel with dirty state | Native `confirm()` prompt (P2.1) — no custom modal until theming lands. |
| Network offline during drag | No mitigation in P2.1 — the Save attempt will surface the error. |

## 4. Testing strategy

1. **Unit — `xml-build.test.ts`**
   - Round-trip: `parseSeagullXml(buildDashboardXml(id, widgets))` reproduces the input structure for a representative dashboard.
   - Special characters (`&`, `<`, `>`, quotes, unicode) in widget titles are correctly escaped.
2. **Unit — `save-dashboard.test.ts`** (mirrors T7 pattern)
   - `saveDashboard(id, widgets)` calls the right URL + body, returns the typed `Dashboard`.
   - Throws `SeagullError` on non-2xx.
3. **Unit — `api-dashboard-put.test.ts`** (mirrors T10 pattern)
   - Route `PUT /api/dashboards/[id]` returns 200 on success, 502 on upstream failure, 422 on Zod validation failure.
4. **E2E — `dashboard-edit.spec.ts`**
   - Start at `/dashboards/1`, click `Edit`, drag CPU line chart from (3,0) to (0,4), click `Save`, land on `/dashboards/1`, reload, confirm the widget is at (0,4).
   - WireMock gets a dynamic mapping for `PUT /dashboards/1.xml` that echoes the submitted body back, so the round-trip is verifiable without real state in the mock.

Total test additions: 3 unit files + 1 e2e. Existing tests (16) remain untouched.

## 6. Risks

| # | Risk | Mitigation |
|---|---|---|
| 1 | Discovery reveals seagull save endpoint uses non-XML body or an entirely different shape than expected | Spike is Task 1. Adapter architecture is a separable file (`xml-build.ts`) easily swapped for a form-encoded builder. |
| 2 | `react-grid-layout` responsive behavior conflicts with our CSS `@theme` token setup or causes SSR hydration warnings | Render only on client (`"use client"` already in place); known issue is addressed by wrapping with `WidthProvider` on mount. |
| 3 | Dirty-state comparison via `deepEqual` is slow for dashboards with many widgets | P2.1 target is the existing fixture scale (3 widgets). If performance becomes an issue, switch to a version counter on each widget mutation. |
| 4 | Concurrent edits on the same dashboard by two users | Not addressed in P2.1; seagull's behavior on concurrent save is unknown. Treated as a P3 concern (surface an `ETag`/`If-Match` story only if the backend supports it). |
| 5 | `react-grid-layout` + touch devices (mobile) | P2.1 targets desktop edit only; the `x/y/w/h` model and library do support touch but the full mobile-edit UX belongs to a later P2 plan. |
