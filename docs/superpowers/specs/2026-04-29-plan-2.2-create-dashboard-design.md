# P2.2 — Create New Dashboard + Widget Palette Design

**Status:** Design approved 2026-04-29 — ready for implementation plan
**Depends on:** P1 MVP (read path), P2.1 Edit (save endpoint + canvas UX)
**Estimate:** ~36 engineering hours

---

## 1. Goal

Let users create a new dashboard from scratch: pick widgets from a catalog, arrange them on the freeform canvas, name the dashboard, and save. Reuse the widget palette in the existing edit flow so widgets can also be added to saved dashboards.

## 2. Scope

**In scope**
- New route: `/dashboards/new`
- Shared `<WidgetPalette>` drawer — mounted on `/dashboards/new` and on `/dashboards/[id]/edit`
- "New Dashboard" button on `/dashboards` list page
- `POST /api/dashboards` endpoint that creates via Seagull and redirects to the created dashboard
- Inline-editable dashboard name on the new-dashboard page (defaults to "Untitled dashboard")
- Hard-coded widget catalog (3 entries matching current WireMock fixtures)

**Out of scope** (deferred)
- Duplicate widgets of the same catalog entry on one dashboard
- Rename or delete existing dashboards
- Dynamic catalog (fetched from backend)
- Inline custom-widget authoring (query editor, series config)
- Custom canvas size (locked to 1920×1080 for now)

## 3. Widget catalog

Hard-coded in `src/config/widget-catalog.ts` as a typed array. Each entry is the template used to seed a new `WidgetRef` when dropped onto the canvas.

```ts
export type WidgetCatalogEntry = {
  id: string;           // matches WidgetRef.id produced on create
  kind: WidgetKind;     // "kpi" | "line" | "table"
  title: string;
  defaultW: number;
  defaultH: number;
};

export const WIDGET_CATALOG: WidgetCatalogEntry[] = [
  { id: "w-cpu-kpi",    kind: "kpi",   title: "CPU %",         defaultW: 260,  defaultH: 160 },
  { id: "w-cpu-line",   kind: "line",  title: "CPU over time", defaultW: 720,  defaultH: 320 },
  { id: "w-hosts-table",kind: "table", title: "Hosts",         defaultW: 1000, defaultH: 320 },
];
```

**Why hard-coded:** the mock API has no catalog endpoint and the three widgets above are the only ones with `WidgetDataSchema` support. Dynamic catalog is a post-MVP concern.

## 4. `WidgetPalette` component

Right-side drawer, always open on `/dashboards/new`, togglable on `/dashboards/[id]/edit`.

**Props**
```ts
type WidgetPaletteProps = {
  catalog: WidgetCatalogEntry[];
  existingWidgetIds: Set<string>;   // for duplicate guard
  onAdd: (entry: WidgetCatalogEntry) => void;
};
```

**Behavior**
- Renders one button per catalog entry (title + kind badge).
- Entry is disabled if `existingWidgetIds.has(entry.id)` — duplicate guard is client-side only.
- Click → `onAdd(entry)` — parent decides where to place it.

**Placement by parent**
New widget lands at `(20, 20)` with the catalog entry's default `w`/`h`. Collisions are allowed (user repositions with react-rnd). First-version trade-off: simple, predictable, user-fixable.

## 5. Creation flow (`/dashboards/new`)

**Initial state**
Local component state holds a `CreateDashboard` (no `id`), not a `Dashboard`. The `EditableDashboardCanvas` only needs `width/height/widgets`, so the missing `id` is a non-issue at the canvas layer.

```ts
const [draft, setDraft] = useState<CreateDashboard>({
  name: "Untitled dashboard",
  owner: "opuser",  // TODO: from session when auth lands
  width: 1920,
  height: 1080,
  widgets: [],
});
```

**UI**
- Inline-editable `<input>` on the header for `name` (same position as read-only title on `/[id]`).
- Reuses `<EditableDashboardCanvas>` with `width/height/widgets` from local state.
- `<WidgetPalette>` on the right.
- Save button top-right; disabled until `widgets.length > 0 && name.trim().length > 0`.

**Save**
- `POST /api/dashboards` with JSON body matching `CreateDashboardSchema` (Dashboard with `id` omitted).
- On 2xx: redirect to `/dashboards/{returned_id}`.
- On error: toast with `SaveDashboardError` message (reuse mapping from P2.1).

## 6. API wiring

### 6.1 Schema
```ts
// src/server/schemas/dashboard.ts
export const CreateDashboardSchema = DashboardSchema.omit({ id: true });
export type CreateDashboard = z.infer<typeof CreateDashboardSchema>;
```

### 6.2 Route
New file: `app/api/dashboards/route.ts`
- `POST` handler: parse body with `CreateDashboardSchema`, call `saveDashboard(input)` (see 6.3), return created dashboard JSON (200) or error mapped via existing `SAVE_ERROR_HTTP` table.

### 6.3 `saveDashboard` branching
Current `saveDashboard(dashboard: Dashboard)` assumes a full dashboard with `id`. Extend to accept create-shape by making `id` optional at the type boundary:

```ts
export async function saveDashboard(
  dashboard: Dashboard | CreateDashboard
): Promise<Dashboard> {
  const body = buildSaveDashboardBody(dashboard);  // omits id when absent
  const raw = await callSeagullJson({ path: SAVE_PATH, body });
  const parsed = SaveResponseSchema.parse(raw);
  if (parsed.output <= 0) throw new SaveDashboardError(parsed.output, ...);
  return getDashboard(String(parsed.output));
}
```

`buildSaveDashboardBody` gains a branch: include `id` only when the input is a `Dashboard` (type narrowing via `"id" in dashboard`). `CreateDashboard` inputs omit the field entirely from the JSON payload. Matches Flex client behavior (create-new sends no `id`).

**Risk flagged:** the create-without-id shape is inferred from the Flex source — not empirically validated against a live OpMon instance. TODO comment mirrors the existing `widgets`-field extension TODO in `saveDashboard`. Must be validated before staging.

## 7. Edit-mode palette

`/dashboards/[id]/edit` gains the same `<WidgetPalette>` to the right of the canvas. Collapsible (keyboard shortcut `p` or a chevron button) since edit page is the "refine existing" flow — palette is secondary there. Defaults to open on first render so the feature is discoverable.

## 8. List-page entry point

`/dashboards/page.tsx`: add a primary "New Dashboard" button in the header next to the page title. Click → `router.push("/dashboards/new")`.

## 11. Risks

1. **Create-without-id payload shape unverified.** Inferred from Flex source; if Seagull rejects a body without `id`, saves will fail with `output: -1`. Mitigation: TODO comment, staging validation before promotion.
2. **Default position (20, 20) stacks widgets.** Users adding multiple widgets quickly will see them overlap. Acceptable for MVP — user drags them apart. Revisit if feedback says it's annoying.
3. **Client-side-only duplicate guard.** If two users open `/dashboards/new` and add the same catalog entry to different dashboards it's fine — the guard is per-instance. No server-side enforcement needed at this layer.
4. **Canvas size locked.** Widgets sized for 1920×1080; if the catalog grows with larger widgets, resize logic will need generalizing.

## 12. Open questions (deferred)

- Should "Save" on an empty canvas be allowed? **Decision:** No — button stays disabled until `widgets.length > 0`. Rationale: avoids zero-widget orphan dashboards; users can always add one throwaway widget.
- Should catalog icons be illustrative (small chart thumbnails) or text-only? **Decision:** Text + kind badge for MVP; icons are visual polish deferred to post-P2.2.
- Where does `owner` come from when auth isn't wired? **Decision:** Hard-coded `"opuser"` to match fixture; TODO for auth integration.
